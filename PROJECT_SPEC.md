# 羽毛球比赛自动分析系统 — 项目规格书

> **版本**: v1.0 | **日期**: 2026-06-01 | **状态**: 开发中

---

## 🎯 项目概述

基于计算机视觉的羽毛球单打比赛自动分析系统。从固定视角拍摄的视频中：

1. **自动检测球场**并建立坐标系
2. **实时追踪**两名球员的脚部位置
3. 根据球员站位**检测比赛状态**
4. 基于羽毛球规则**自动计分**
5. 生成**带可视化覆盖层的标注视频**

**核心原则**: 系统**不识别羽毛球或击球动作**，仅通过球员位置变化和比赛规则推断比分。

## 🧩 模块详细说明

### Module 1: 球场校准器 ([calibrator.py](calibrator.py))

| 属性 | 说明 |
|------|------|
| **职责** | 单次处理，检测球场边界并建立坐标系 |

**任务**:

1. **自动检测**（默认模式）
   - Canny边缘检测 + HoughLinesP检测直线
   - 从直线中识别所有可能的四边形场地（可能有多个场地在画面中）
   - **多场地策略**:
     - 检测所有候选场地（不限制数量）
     - 选择**面积最大**的场地作为目标
     - 自动判定场地类型：
       - 单打场地: 13.4m × 5.18m (宽长比 ≈ 0.386)
       - 双打场地: 13.4m × 6.10m (宽长比 ≈ 0.455)
       - 判定方法: 透视校正后计算宽长比，选择最接近的场地类型
     - 如果判定为双打场地，区域定义使用双打尺寸
   - **不完整场地处理**:
     - 只需**3条边**即可推断第4条边
     - 利用已知的场地比例约束（矩形，已知长宽比）
     - 通过3条边的端点 + 矩形约束反推第4个角点
     - 4条边的场景: 直接使用4个交点
     - 3条边的场景: 用平行四边形/矩形约束补全
     - 2条边及以下: 自动检测失败，回退手动模式
   - 提取4个角点：左上、右上、右下、左下

2. **透视变换**
   - `cv2.getPerspectiveTransform()` 建立变换矩阵
   - 将任意角度球场 → 标准俯视图（宽度按场地类型自适应）
   - 单打: 800×400, 双打: 940×400（保持比例）
   - 保存变换矩阵 + 场地类型供后续使用

3. **区域定义**
   - 根据判定出的场地类型定义4个发球区:
     - **单打发球区** (宽5.18m):
       - 右近发球区 (0-2m, 1.5-3.5m)
       - 左近发球区 (0-2m, 1.5-3.5m)
       - 右远发球区 (11.4-13.4m, 1.5-3.5m)
       - 左远发球区 (11.4-13.4m, 1.5-3.5m)
     - **双打发球区** (宽6.10m, 发球区更宽):
       - 右近发球区 (0-2m, 1.98-3.96m)
       - 左近发球区 (0-2m, 1.98-3.96m)
       - 右远发球区 (11.4-13.4m, 1.98-3.96m)
       - 左远发球区 (11.4-13.4m, 1.98-3.96m)
   - 区域容差: 0.3m

4. **手动回退**
   - 自动检测失败时，GUI手动标注4个角点
   - 显示原视频帧，等待用户点击
   - 手动标注后也自动判定单打/双打

**输入**: 视频第一帧  
**输出**: 透视变换矩阵、场地类型(singles/doubles)、标准球场尺寸、发球区坐标

### Module 2: 球员追踪器 ([tracker.py](tracker.py))

| 属性 | 说明 |
|------|------|
| **职责** | 逐帧处理，检测并追踪两名球员的脚部位置 |

**任务**:

1. **球员检测**
   - YOLOv8-pose模型 (`yolov8n-pose.pt`) 检测人体关键点

2. **脚部定位策略**（优先级顺序）
   - 首选: 双脚踝中点 (COCO: left_ankle=15, right_ankle=16)
   - 次选: 单脚踝位置（一脚不可见时）
   - 备选: 检测框底部中心点

3. **坐标转换**
   - 应用透视变换矩阵
   - 像素坐标 → 球场坐标系（米）
   - 标准化坐标范围 (0-1)

4. **身份保持**
   - 第0帧: 画面下方球员为A，上方为B
   - 后续帧: 最近邻追踪，保持A/B一致性
   - 最大丢失帧数: 30帧

5. **坐标平滑**
   - 3帧滑动平均滤波器
   - 减少检测抖动

**输入**: 视频帧、透视变换矩阵  
**输出**: 每帧两名球员的标准坐标 (A, B)

### Module 3: 状态检测器 ([detector.py](detector.py))

| 属性 | 说明 |
|------|------|
| **职责** | 滑动窗口检测，判断比赛状态 |

**任务**:

1. **位置检测**: 实时计算两名球员所在发球区
2. **运动检测**: 速度 = Δ位置 / Δ时间，阈值 0.3m/s
3. **事件触发**:
   - 就绪条件: 两人都在正确区域 + 速度<阈值
   - 持续时间: 连续3秒满足条件
   - 输出: `ReadyEvent(timestamp, player_a_zone, player_b_zone)`
4. **冷却机制**: 事件触发后5秒冷却期，防止重复触发

**输入**: 球员位置序列、发球区定义  
**输出**: 就绪事件序列

---

### Module 4: 比分计算器 ([scorer.py](scorer.py))

| 属性 | 说明 |
|------|------|
| **职责** | 状态机，根据比赛规则计算比分 |

**任务**:

1. **初始化**: first_server决定发球方，初始比分 0:0
2. **得分规则**（21分制）:
   - 发球方得分 → 发球方保持发球权，比分+1
   - 接发方得分 → 交换发球权，接发方比分+1
3. **发球区规则**:
   - 偶分数 → 右发球区发球
   - 奇分数 → 左发球区发球
4. **位置验证逻辑**:
   - 情况A: 都在正确区 → 正常比赛，不发分
   - 情况B: 发球方换了左/右区 → 发球方得分
   - 情况C: 发球方变了 → 接发方得分（换发）
5. **特殊规则**: 11分换边、20:20平分需净胜2分、29:29下一分获胜

**输入**: 就绪事件序列  
**输出**: 比分更新记录

---

### Module 5: 覆盖渲染器 ([overlay.py](overlay.py))

| 属性 | 说明 |
|------|------|
| **职责** | 可视化渲染，创建标注视频 |

**任务**:

1. **浮动窗口**: 右上角 250×500px, 透明度0.6
2. **迷你球场图**: 绿底白线 + A(红点)/B(蓝点)实时位置
3. **比分显示**: `A 12 : 8 B`，当前发球方标记 `[发]`
4. **轨迹显示**: 最近20帧移动轨迹，A=橙色, B=青色
5. **额外输出**: `annotated.mp4`, `scores.json`, `score_curve.png`

**输入**: 原视频、球员位置、比分序列  
**输出**: 标注视频 + 数据文件

### Module 6: 主控制器 ([main.py](main.py))

| 属性 | 说明 |
|------|------|
| **职责** | 串联整个流程，处理命令行参数 |

**参数**:
```
--video          输入视频路径（必需）
--first-server   首局发球方 A/B（默认A）
--output-video   输出视频路径（默认annotated.mp4）
--output-scores  比分JSON路径（默认scores.json）
--manual-calibration  手动标注球场角点
```

**流程**: 读取视频 → Calibrator → 逐帧(Tracker→Detector→Scorer→Overlay) → 保存输出

## 📊 验收标准

| 指标 | 目标 |
|------|------|
| 稳定性 | 处理10+分钟视频不崩溃 |
| 球员定位误差 | < 0.5米 |
| 最终比分误差 | ≤ ±3分 |
| 处理速度 | > 10fps (640×480) |
| 鲁棒性 | 应对遮挡、光照变化、不同球场 |

---

## ⚠️ 限制条件

- 固定机位拍摄，不移动/变焦
- 包含完整单打球场
- 球场线清晰可见
- 分辨率 ≥ 720p, 帧率 ≥ 25fps
- 仅支持单打比赛
